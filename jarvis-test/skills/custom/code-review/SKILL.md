# Скилл: Анализ кода (Code Review)

## Когда активировать
Клиент просит: "проверь код", "сделай ревью", "найди баги", "что не так с кодом", "проанализируй файл/проект"

## Контекст клиента
- Стек: React, Next.js, Vanilla JS, Vite, Supabase, Telegram WebApp
- Хостинг: Vercel
- Бэкенд: Python/aiohttp, Supabase Edge Functions (Deno)
- Дизайн: тёмная тема, glassmorphism
- Аудитория: русскоязычная

## Процесс анализа

### 1. Сбор контекста
- Прочитай файл(ы) которые нужно проверить
- Посмотри соседние файлы для понимания архитектуры
- Проверь package.json / requirements.txt на зависимости

### 2. Чеклист проверки

**Безопасность (приоритет 1):**
- XSS: экранирование пользовательского ввода, innerHTML vs textContent
- SQL-инъекции: параметризованные запросы в Supabase
- Утечка ключей: нет ли API-ключей в коде (должны быть в .env)
- RLS: правильность Row Level Security политик
- CORS: настройки для Edge Functions
- Telegram initData: валидация подписи на сервере

**Архитектура (приоритет 2):**
- Разделение ответственности (UI / логика / данные)
- Дублирование кода — предложи вынести в утилиту только если 3+ повторений
- Размер файлов — если > 300 строк, предложи разделить
- Зависимости — нет ли лишних или устаревших

**Производительность (приоритет 3):**
- Лишние ре-рендеры (React: useCallback, useMemo где нужно)
- Тяжёлые операции в цикле рендера
- Размер бандла (ненужные импорты)
- Оптимизация запросов к Supabase (select конкретных полей, пагинация)
- Lazy loading для изображений и тяжёлых компонентов

**Читаемость (приоритет 4):**
- Понятный naming (переменные, функции, файлы)
- Консистентность стиля в рамках проекта
- Магические числа — именованные константы
- Сложные условия — вынести в функцию с говорящим именем

**Telegram-специфичное:**
- Корректное использование WebApp SDK (tg.expand(), tg.close(), MainButton)
- Обработка тем (themeParams) для нативного вида
- Haptic feedback где уместно
- BackButton поведение

## Формат ответа

Результат в виде секций по приоритету. Только реальные проблемы, не придирки.

Формат каждого замечания:
- [!] критично — баг, уязвимость, потеря данных
- [~] важно — влияет на UX или производительность
- [·] мелочь — стиль, naming, можно отложить

В конце — общая оценка: что хорошо, что главное исправить.

## Чего НЕ делать
- Не придираться к отсутствию TypeScript если проект на JS
- Не предлагать рефакторинг ради рефакторинга
- Не добавлять комментарии/JSDoc если код и так понятен
